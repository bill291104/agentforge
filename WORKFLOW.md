# AgentForge 4단계 워크플로우 명세

> 사용자가 2026-04-27 정의한 원문 기준.
> 구현 상태와 대조하여 갭을 추적한다.

---

## 1단계: 요구사항 정의

- 리더 에이전트가 사용자 요구사항을 몇 차례 대화를 통해 수집.
- 리더 에이전트가 요구사항을 구체화할 만큼 충분히 대화했다고 판단하면 요구사항을 우선 구체화함.
- 구체화한 요구사항을 사용자에게 보여주고 승인을 받을 때까지 요구사항 정의 수정과 보고를 반복.
- 요구사항 정의를 승인 받았다면 이를 문서화하고 작업세션과 작업공간을 생성하여 문서를 저장하고 계획 수립 단계로 넘어감.

**구현 상태**: ✅ `ClarifierAgent` → `clarification_node` → 요구사항 확정 → 세션 생성 흐름 동작

---

## 2단계: 계획 수립

- 문서화된 요구사항을 충족하기 위한 계획을 세움. 계획을 세우기 위해서는 다음과 같은 것들을 고려해야 함:
  - 사용 가능한 하위 에이전트 풀
  - 작업의 복잡도와 이를 수행 가능하다고 판단되는 모델
  - 작업의 우선순위 및 의존도를 파악해 병렬로 진행 가능한 작업이 있는지
- 작업을 어떻게 분리했으며 어떤 모델의 에이전트에게 어떻게 분배하고 병렬 수행이나 작업 순서를 어떻게 할 것인지 모두 포함된 문서를 작성하고 작업공간에 저장.
- 저장된 문서를 사용자에게 전달 (Slack).
- 사용자가 계획서를 승인할 때까지 계획서 수정과 전달을 반복.
- 계획서를 승인받았다면 작업 수행 단계로 넘어감.

**구현 상태**: ✅ `add_task`/`submit_plan` 멀티-툴 루프 → `present_plan_node` → PLAN.md → 승인/수정 버튼

---

## 3단계: 작업 수행

- 계획서에 정의된 대로 하위 작업을 수행. 리더 에이전트는 하위 작업 하나당 다음 3단계를 거침:

  **a. 지시**: 목표와 완료 평가 기준이 정의된 작업 지시서를 작성하고 저장함.
  해당 작업을 수행할 독립적인 공간(git 브랜치)을 마련.
  작업을 수행할 에이전트를 생성하여 적절한 프롬프트와 함께 작업 지시서를 참조하여 입력.

  **b. 대기**: 하위 에이전트에게 작업을 지시하고 보고가 올 때까지 기다림.
  기다리면서 병렬로 지시 가능한 작업이 있다면 작업 수행 단계를 병렬로 실행 가능.

  **c. 평가**: 하위 에이전트가 작업을 마치면 작업 지시서에 정의했던 평가 기준을 사용하여
  작업이 완수되었는지 평가. 완수되지 않았다면 작업 지시서를 보완하여 위 단계를 반복.

- 하위 작업이 하나 완료될 때마다 해당 PR을 main 브랜치에 병합하고
  계획서에 세부 단계에 완료 표시를 하고 특이 사항이 있다면 추가하여 사용자에게 보고 (Slack).
- 모든 계획서에서 정의한 모든 하위 작업이 완료되면 보고 단계로 넘어감.

**구현 상태**:
- ✅ `dispatch_workers_node` → 병렬 워커 실행
- ✅ `verify_ci_node` + `verify_semantic_node` → 이중 검증
- ✅ `merge_task_node` → main 브랜치 병합 + PLAN.md 완료 표시
- ✅ L0~L4 에스컬레이션 프로토콜 (자동 재시도 → 모델 업그레이드 → 사용자 개입)
- ⚠️ `interrupt_l4` "계속 진행" 선택 시 태스크가 PENDING으로 리셋되지 않음 → **버그 #1 수정 대상**

---

## 4단계: 보고

- 계획서에 정의한 모든 하위 작업이 완료되면 통합테스트나 최종 검수를 수행. 여기서 문제를 발견하면 이전 단계로 돌아감.
- 문제가 없다면 작업이 어떻게 수행되었고 어디서 몇 번 재시도 했으며 얼만큼의 토큰을 사용했는지
  그리고 결과물을 사용자는 어떻게 확인 (사용자 테스트) 할 수 있는지 방법을 포함하여
  최종 보고서를 작성하고 작업 디렉토리에 저장.
- 최종 보고서를 사용자에게 전달 (Slack).
- 사용자는 보고서를 확인하고 검수 후 작업물이 원하는 대로 나왔는지 판단.
- 부족한 부분이나 문제점이 있다면 사용자는 이를 리더 에이전트에게 전달 (Slack) 하고
  리더 에이전트는 요구사항 정의 단계로 돌아감.

**구현 상태**:
- ✅ `finalize_node` → FINAL_REPORT.md 작성 (재시도 횟수, 토큰, 실행 방법 포함)
- ✅ `_detect_run_guide()` → package.json / requirements.txt 등 파일 기반 실행 방법 안내
- ✅ `run_tests()` → 통합 테스트 실행 (Docker 우선, 로컬 폴백) + 실패 시 check_context 복귀

## 공통 제약

- 모든 단계에서 사용자는 리더 에이전트에게 전체적인 작업세션이나 세부 작업을 중지하거나
  재개하라고 지시할 수 있으며, 이러한 지시를 받은 에이전트는 해당 작업 세션이 무엇이었는지
  또는 사용자의 지시에 따라 완전히 새로운 세션 또는 하위 작업을 생성해야 하는지 판단해야 함.

- 모든 단계에서 AF가 리로드 되면 사용자의 재개하라는 지시를 통해 해당 작업 세션이 무엇이었는지
  찾아서 작업 디렉토리를 확인해 어디까지 진행되었는지 스스로 파악하고 멈춘 부분부터 재개할 수 있어야 함.

**구현 상태**: ✅ `_restore_sessions()` + LangGraph checkpoint 기반 재개 + SQLite 스레드 영속화
